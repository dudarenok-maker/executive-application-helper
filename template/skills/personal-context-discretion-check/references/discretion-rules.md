# Personal context discretion rules — canonical reference (rev 2026-06-12, codifies the the framework personal-context discretion convention)

## The rule

Sensitive personal information shared by [Candidate First Name] for a specific deliverable is used **only in that named deliverable**, with his explicit consent, and is **not saved to**:

- The evidence bank — no entry, no proof point, no caution row
- The memory system — no memory file, no memory entry
- Any future deliverable without his explicit direction at the start of that session

Each surfacing is **one-time and bounded**. If [Candidate First Name] wants the same context in a future deliverable, he says so explicitly at that session's kickoff; nothing in the current session may pre-commit it.

## What counts as sensitive

- Health, medical history, diagnoses, mental-health details
- Family circumstances beyond what the bank's personal-content section already carries with consent
- Protected attributes (the system sensitive-personal-information list)
- Anything [Candidate First Name] flags as personal when sharing it

When in doubt, treat as sensitive and check.

## Check mechanics

1. **Identify the context and its key terms** from the session record (e.g. a condition name, a family circumstance, a date or place that identifies it).
2. **Consent scope:** find the candidate's explicit direction tying the context to the named deliverable. Use in any other artefact is out of scope regardless of how natural it seemed.
3. **Bank boundary:** grep every bank file touched this session (and the changelog entry being drafted) for the key terms. The changelog may record the *category* ("lived-experience response prepared for Q1") but never the substance.
4. **Memory boundary:** check any memory files created or amended this session for the key terms.
5. **Forward-staging boundary:** check session notes, templates, "next steps" lists and the maintenance report for anything that stages the context for future reuse.

## On finding a propagation

Reverse it **in-session**: remove the content from the file, re-run the check, and only then close. Never park a propagation finding as a pending follow-up — the whole point of the boundary is that the context does not persist anywhere it wasn't consented to.

## Calibration standard

A clean handling looks like: a sensitive detail used with explicit direction in one named brief, and deliberately not propagated to `Examples_Section_D_E_Personal_Maintenance.md` or any memory file.
